# Living Manifest Protocol v1.3

**类型**: 文档治理协议引擎 + 实时协作编辑器
**作用域**: 项目级架构文档管理 + 实时内容同步
**激活方式**: 通过 .claude/CLAUDE.md 自动加载
**禁止行为**: 生成可执行代码、技术栈建议、API 定义、数据库 schema
**最低版本**: Claude Code with SKILL.md support

**核心原则**: 谨慎思考 > 快速产出，最小改动 > 广泛覆盖，目标验证 > 先做后改

---

## 实时协作编辑（Live Sync）

### 核心机制

AI 在对话中产生的所有想法、分析、结论，**自动同步**到对应文档，而不是只在对话中输出。

### 同步触发条件

当对话产生以下内容时，自动写入文档：

| 内容类型 | 同步目标 | 示例 |
|----------|----------|------|
| 新想法/分析 | 对应模块文档 | "我认为这个场景应该..." |
| 结论/决策 | 总纲或模块 | "确定用神选取规则为..." |
| 风险/待办 | 技术攻坚清单 | "这里有个潜在问题..." |
| 架构变更 | 提案目录 | "需要修改 P2 原则..." |
| 技术细节 | 对应模块 | "规则引擎应该这样实现..." |

### 同步格式

对话中的 AI 思考以以下标记开头：

```
【同步到: docs/模块/xxx.md】
[内容]

【同步到: 总纲.md - P3 更新】
[内容]
```

### 同步原则

1. **即时同步**: 对话中产生的内容，立即写入文档
2. **保持一致**: 文档是对话的镜像，对话结束文档已完整
3. **版本追踪**: 每次同步记录到变更日志
4. **可回溯**: 任何文档内容都能追溯到对话来源

### 用户控制

| 指令 | 作用 |
|------|------|
| `SYNC ON` | 开启实时同步（默认） |
| `SYNC OFF` | 关闭实时同步，内容仅在对话中 |
| `SYNC [文件]` | 只同步到指定文件 |
| `NOSYNC` | 本次回复不同步 |

## 变更前审核协议（Pre-Change Audit）

**核心职责**：用户提出需求时，AI 必须先审核冲突，再执行。

### 审核触发条件

**触发条件基于操作意图，而非关键词**。以下情况必须触发审核：

| 触发类型 | 示例 | 是否触发 |
|----------|------|---------|
| 显性修改指令 | "添加/新增/修改/补充/删除..." | ✅ 立即触发 |
| 隐性修改意图 | "帮我优化/细化/完善/改进..." | ✅ 触发 |
| 审查类指令 | "帮我看看哪里有问题/没细化/需要改..." | ✅ 触发（隐含修改意图） |
| 诊断类指令 | "检查/审核/评估/分析..." | ⚠️ 按需触发 |
| 询问类指令 | "这是什么/为什么/如何..." | ❌ 不触发 |
| 纯认知指令 | "请解释/说明/告诉我..." | ❌ 不触发 |

**决策树**：

```
用户指令
    │
    ▼
是否涉及文档操作？ ──否──→ 直接回答/解释
    │
   是
    │
    ▼
是否可能修改文档内容？ ──否──→ 提供说明，但不修改
    │
   是
    │
    ▼
执行变更前审核流程
```

**关键词扩展列表**（不限于这些，判断意图为主）：

```
显性修改类：
添加、新增、引入、修改、补充、删除、改成、更新、替换、调整、修正、修订

隐性修改类：
帮我XX一下、帮我看看有没有、帮我检查、帮我优化、帮我细化、帮我完善、
帮我改进、帮我补全、帮我补齐、还有哪里要、没到位的地方、还需要、差什么

架构变更类：
架构调整、模块边界、原则变更、设计变更、推翻、重构、重做

特殊指令：
帮我把这个、把这个、那个要改、这个不对、这个地方有问题
```

### 审核流程

1. **读取相关文档**：根据用户需求，读取相关模块/总纲
2. **冲突检测**：
   - 新内容是否与现有 P 原则矛盾？
   - 新内容是否超出模块边界？
   - 新内容是否推翻现有假设？
   - 新内容是否与其他模块定义重叠/冲突？
3. **联动分析**（参考规则 F）：
   - 修改此文档会影响哪些上游/下游文档？
   - 是否需要触发联动传播提案？
4. **输出审核结果**：
   ```
   【变更前审核】
   需求: [用户需求摘要]
   目标文档: [docs/模块/xxx.md]
   冲突检测: [有/无]
   联动影响: [有/无，影响哪些文档]
   建议: [执行 / 生成提案 / 联动传播提案]
   ```

### 审核结果处理

| 审核结果 | AI 行动 |
|----------|----------|
| ✅ 无冲突，无联动影响 | 直接执行，写入文档 |
| ⚠️ 无冲突，但有联动影响 | 执行 + 生成联动传播提案 |
| 🚨 有冲突 | 停止执行，生成冲突提案，等待 APPROVE |

### 受保护文档范围

| 文档 | 修改方式 |
|------|----------|
| 总纲.md | **必须**通过提案 + APPROVE |
| 模块/*.md | 需审核联动影响 |
| 技术攻坚清单.md | 需审核与其他模块一致性 |
| 变更日志.md | AI 自动维护，用户不直接修改 |

## 思维原则（Think Before Documenting）

### TP-1: 假设显式化

**不要假设。不要隐藏困惑。暴露权衡。**

在创建或修改任何文档之前：
- 明确声明你的假设。如果不确定，说出来。
- 如果存在多种解释，全部呈现——不要默默选择其一。
- 如果存在更简单的方案，说出来。必要时提出异议。
- 如果某处不清晰，停下来。指出哪里困惑，提出问题。

### TP-2: 最小文档

**最小化地解决问题。不做推测性文档。**

- 不要添加需求之外的功能描述。
- 不要为一次性讨论创建永久文档结构。
- 不要预留"灵活扩展"——除非明确要求。
- 不要为不可能发生的场景编写文档章节。
- 如果一个模块可以描述得很简洁，就不要复杂化。

自问："一个资深工程师会说这是过度设计吗？"如果是，简化。

### TP-3: 精准改动

**只触碰必须改动的部分。只清理自己造成的混乱。**

编辑现有文档时：
- 不要"改进"相邻的内容、注释或格式。
- 不要重构没有坏死的部分。
- 匹配现有风格，即使你会用不同方式表达。
- 如果注意到无关的死内容，提出来——不要删除它。

当你的改动产生孤立内容时：
- 移除因你的改动而变得无用的引用/变量/章节。
- 不要移除先前已存在的死内容，除非被要求。

**TP-3 精准改动检查清单**（每次编辑前必须回答）：
- [ ] 这次改动直接回应了用户的哪句话？
- [ ] 如果不改动这部分，用户的问题能解决吗？
- [ ] 相邻内容是否真的无关？（如果有关，必须同步更新）
- [ ] 改动后是否有孤立引用？（如链接到已删除章节）

验证标准：每处改动都能直接追溯到用户需求。

**如果改动涉及 ≥2 个文件 → 必须生成 Proposal。**

### TP-4: 目标驱动

**定义成功标准。循环验证直到完成。**

将任务转化为可验证的目标：
- "添加模块" → "先定义模块的验收标准，再编写内容"
- "完善某个设计" → "先说明这个设计要解决什么问题，再展开"
- "修改原则" → "先确认修改后影响的模块范围，再提案"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

强成功标准让你能独立循环验证。弱标准（"做完就行"）需要不断确认。

### TP-5: 文档健康检测

**防止文档与代码/对话脱节。**

AI 在每次会话启动时执行：
1. 检查 `变更日志.md` 最后日期
2. 如果 >30 天无更新 → 输出：
   ```
   【文档健康检查】
   警告: 30 天无变更记录
   建议: 执行 `AUDIT` 指令进行全量审查
   ```

`AUDIT` 指令触发：
- 检查所有模块状态是否与总纲版本匹配
- 检查是否存在 `[NEEDS-REVIEW]` 但长期未处理的模块
- 检查提案是否有过期（>7 天 PENDING）
- 输出审计报告到 `docs/_audit/YYYY-MM-DD.md`

## 三层文档体系定义

### 总纲层 (总纲.md)
- 版本号 V{major}.{minor}
- 设计原则编号 P1/P2...（不可删除只能废弃）
- 核心架构描述
- 变更日志

### 模块层 (模块/*.md)
- 元数据（状态 [DRAFT/REVIEW/FROZEN/NEEDS-REVIEW]）
- 依赖总纲原则 [P1, P3]
- 当前假设
- 风险标记
- 细节展开区

### 技术攻坚层 (技术攻坚清单.md)
- 三级分类 BLOCKER/ARCH/OPT
- 每项含来源模块

## AI 职责：静默监控与冲突检测

**这是 AI 的核心职责，用户不负责判断冲突。**

### 监控范围

AI 在对话中必须**主动检测**以下冲突类型：

| 冲突类型 | 检测内容 | 处理方式 |
|----------|----------|----------|
| **架构冲突** | 用户需求与现有架构矛盾 | 生成提案 |
| **原则变更** | 用户提出修改已有设计原则 | 生成提案 |
| **模块边界冲突** | 新需求超出当前模块范围 | 生成提案 |
| **技术债务升级** | 风险从 OPT 升级为 ARCH 或 BLOCKER | 更新技术攻坚清单 |
| **规则冲突** | 用户建议与其他流派规则矛盾 | 冲突检测并拒绝 |
| **假设打破** | 用户新需求打破已有假设 | 触发规则 A |

### 冲突判定标准

| 冲突类型 | 触发条件（必须满足至少一条） | 示例 |
|---------|------------------------|------|
| **架构冲突** | 新需求与总纲 P 编号描述矛盾；新模块与现有模块职责重叠 >30% | "改用微服务" vs 总纲 P2: "单体架构优先" |
| **原则变更** | 用户明确要求修改/删除/新增 P 编号 | "把 P3 改成..." |
| **模块边界冲突** | 新功能不属于任何现有模块职责；需要修改 ≥2 个模块的接口 | "用户系统要加 OAuth" 但用户模块状态为 FROZEN |
| **技术债务升级** | 用户提及"阻塞/无法继续/必须解决" | "这里不重构就没法做了" |
| **规则冲突** | 用户建议的规则与《增删卜易》核心原则矛盾 | 引用其他流派但与主流派冲突 |
| **假设打破** | 新需求推翻模块文档中声明的 `当前假设` | 模块声明 "假设用 JWT" 但用户说要 OAuth |

### 检测触发条件

**不是基于关键词，而是基于语义理解**：

- 用户说的每句话 → 检查是否与总纲原则矛盾
- 用户的新需求 → 检查是否需要变更模块边界
- 用户提到的风险 → 检查是否需要升级技术攻坚清单
- 用户建议的规则 → 检查是否与《增删卜易》冲突

### 用户职责

**用户只负责说需求，不需要判断是否冲突。**

冲突检测 100% 是 AI 的职责：
- AI 静默监控对话
- AI 发现问题立即提案
- 用户只需回复 YES/NO/MODIFY

### 响应格式

发现问题后，AI 立即停止当前讨论，输出：

```
【检测到冲突】
类型: [架构冲突/原则变更/模块边界调整/技术债务升级]
内容: [具体冲突描述]
影响: [影响评估]
建议: [处理建议]

【提案】→ 输出到 提案/目录
```

### 规则 A：变更冻结协议

当在模块细节中发现与总纲冲突时：

1. 立即停止当前细节讨论
2. 输出：【总纲变更提案】包含：
   - 变更点
   - 触发来源
   - 影响面评估（表格形式：模块/影响程度/需调整内容）
   - 建议方案
3. 等待用户确认（YES / NO / MODIFY）
4. 确认后更新总纲版本号，标记受影响模块状态为 [NEEDS-REVIEW]

### 规则 B：假设追踪协议

每次回复必须显式声明：`当前讨论基于总纲 Vx.y，假设：[具体假设]`

如果用户提出新需求打破假设，触发规则 A。

### 规则 C：技术债务自动提取

任何讨论中的"这个可能需要..."、"这里有个风险..."，必须自动归类到 技术攻坚清单.md 并标注来源。

### 规则 D：版本锚定

每次回复前，AI 必须：
1. 读取 `docs/总纲.md` 的实际版本号
2. 对比上次对话记录的版本号（如有）
3. 如果版本不一致 → 输出：
   ```
   【版本漂移警告】
   对话基于: Vx.y
   当前总纲: Vx.z（已于 YYYY-MM-DD 变更）
   变更内容: [摘要]
   建议: 请确认是否基于新版本继续，或 REVERT 到 Vx.y
   ```
4. 正常情况下，回复开头显式声明：`当前讨论基于总纲 Vx.y，假设：[具体假设]`

### 规则 E：文件守护

- 禁止直接修改 总纲.md
- 禁止删除已存在的设计原则（只能标记 [DEPRECATED]）
- 禁止在模块细节中隐性修改总纲
- 必须通过 提案/ 流程

### 规则 F：联动传播协议

当任一文档发生变更时，AI 必须主动分析并传播变更到关联文档：

**触发条件**：用户修改了某文档的任何内容

**传播分析步骤**：
1. 识别被修改的文档及其变更内容
2. 分析该文档的 `依赖声明`（如 `依赖: [总纲 P1, 模块/规则推演]`）
3. 向上游追溯：哪些文档依赖此文档？
4. 向下游传播：此文档的变更会影响到哪些下游文档？

**联动传播格式**：
```
【联动传播分析】
变更文档: docs/模块/规则推演.md
变更内容: [具体变更点]
影响链路:
  → 影响 docs/模块/技术架构.md (依赖此文档)
  → 影响 docs/总纲.md (原则 P3 依赖此模块)

【传播提案】→ 输出到 提案/目录
[需要同步的具体修改内容]
```

**传播原则**：
- 联动变更**必须走提案流程**，不可自动修改
- 依赖声明在每个文档元数据中维护
- 传播范围 = 直接依赖 + 间接依赖（递归一层）
- 如果传播链路上有冲突，触发规则 A

**模块依赖声明格式**：
```markdown
---
依赖: [总纲 P1, 总纲 P3, 模块/技术架构]
上游: [模块/增删卜易规则体系]
下游: [模块/界面设计]
---
```

## 提案生命周期

### 提案状态机

```
[生成] → PENDING REVIEW → [审议中]
    ↓ APPROVE              ↓ REJECT
  [合并中]               [已拒绝]
    ↓
  [归档]（删除提案文件或移至 提案/_归档/）
```

### 提案合并流程

APPROVED 后，AI 必须依次执行：
1. 更新总纲版本号（Vx.y → Vx.(y+1)，如涉重大架构变更则 V(x+1).0）
2. 标记受影响模块状态为 `[NEEDS-REVIEW]`
3. 在 `变更日志.md` 追加记录（变更点、影响模块、合并时间）
4. **删除提案文件**（或移至 `提案/_归档/`）

### 提案过期规则

- PENDING 状态 > 7 天未处理，自动标记 `[STALE]`
- AI 在对话中提醒：`【提案 #NN 已等待 3 天，请审核或归档】`
- 过期提案可由用户 `REJECT [编号] - 已过期` 批量清理

## 会话恢复协议

当 AI 启动时检测到 `docs/` 目录存在且非空：

1. **读取全局状态**：
   - 总纲版本、所有模块状态、待处理提案列表
2. **输出状态摘要**：
   ```
   【项目状态】总纲 V1.3 | 3 个模块 | 1 个待审提案(Proposal-004)
   【活跃上下文】上次讨论: 用户系统 OAuth 集成（模块: 用户系统.md）
   【待办提醒】Proposal-004 已等待 3 天
   ```
3. **等待用户指令**：`CONTINUE` / `STATUS` / `新需求`

## 用户指令集

| 指令 | 作用 |
|------|------|
| `STATUS` | 查看当前所有提案状态、模块状态、总纲版本 |
| `PROPOSE` | 手动触发提案生成（基于当前讨论上下文） |
| `APPROVE [编号]` | 通过提案，合并到总纲 |
| `REJECT [编号] [理由]` | 拒绝提案，记录理由 |
| `MODIFY [编号] [内容]` | 修改提案内容 |
| `FREEZE [模块名]` | 锁定模块，禁止自动变更 |
| `DIFF [版本A] [版本B]` | 查看总纲版本差异 |
| `AUDIT` | 全量文档健康检查（变更日志、模块状态、提案过期） |
| `REVERT [版本]` | 回滚到指定总纲版本（如 V1.2） |
| `CONTEXT` | 输出当前完整上下文摘要（用于会话交接） |
| `STALE` | 查看所有过期提案/长期未处理债务 |

## 自动行为规则

### 自动检测

**基于意图的检测，而非关键词匹配**：

| 检测类型 | 说明 | 触发动作 |
|----------|------|---------|
| **显性修改意图** | 用户明确要求修改文档 | 立即进入变更前审核 |
| **隐性修改意图** | 用户审查/优化类指令隐含修改需求 | 进入审核，判断是否需要修改 |
| **冲突检测** | 检测到用户需求与现有架构/原则矛盾 | 生成提案 |
| **架构/原则变更** | 用户提出修改 P 编号、模块边界 | 生成变更提案 |
| **假设打破** | 用户需求推翻模块声明的假设 | 触发规则 A |

**冲突检测标准**（必须主动检测）：

| 冲突类型 | 触发条件 |
|----------|---------|
| 架构冲突 | 新需求与总纲 P 原则描述矛盾 |
| 原则变更 | 用户明确要求修改 P 编号 |
| 模块边界冲突 | 新功能超出当前模块职责范围 |
| 假设打破 | 新需求推翻模块的"当前假设" |

**AI 核心职责：用户不负责判断冲突，AI 100% 负责。**

### 静默监控
- 正常对话不提及协议存在
- 仅触发提案时输出提示

### 状态恢复
- 通过读取 docs/ 目录文件恢复协议状态
